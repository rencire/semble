# ADR-0003: Add v0.3.0 project overlays to `mkFlake`

## Status

Accepted

## Context

Semble consumers can already define local overlays for flake `packages` and
`devShells`, but those overlays are not automatically applied to the `pkgs`
instance used for `nixosConfigurations` and image builds.

That creates an easy-to-miss split:

- `pkgs` in flake-level outputs can see project overlays
- `pkgs` in host and image module evaluation cannot, unless the consumer adds a
  separate `nixpkgs.overlays` module manually

This breaks the expectation that a project-local package made available through
an overlay can be used uniformly across:

- local flake outputs
- host modules
- image modules

Consumers should not need to duplicate overlay application logic outside the
Semble API just to make host evaluation see the same project package set.

## Decision

Semble adds an optional `overlays` argument to `lib.mkFlake`:

```nix
inputs.semble.lib.mkFlake {
  inherit inputs;
  root = ./.;
  overlays = [
    (import ./overlays/default.nix)
  ];
}
```

Semble applies these overlays to the NixOS module evaluation path by appending a
small internal module that sets:

```nix
nixpkgs.overlays = overlays;
```

This applies to:

- `nixosConfigurations`
- `images`

## Consequences

### Positive

- Consumers define project overlays once at the Semble entrypoint.
- Host and image module evaluation can use overlay-provided packages directly
  through `pkgs`.
- Overlay behavior becomes part of the Semble consumer API instead of an
  external workaround.

### Neutral

- Consumers that also create custom flake `packages` and `devShells` still need
  to apply overlays in those outputs themselves. Semble only owns host and image
  evaluation.

### Negative

- Overlay ordering is now part of the public Semble API surface and must remain
  stable.

## Notes

- This is intentionally the minimal API shape for v0.3.0.
- Semble does not yet standardize overlay file discovery such as
  `./overlays/*.nix`; consumers pass the list explicitly.
