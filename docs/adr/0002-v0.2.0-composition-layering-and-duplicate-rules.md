# ADR-0002: Adopt v0.2.0 composition layering and duplicate rules

## Status

Accepted

## Date

2026-03-10

## Context

After introducing host-level `modules` and `inputModules`, the next design
question was how to keep the layering model simple without forcing consumers
into premature abstraction. The API needed a clear answer for:

- when to use `inputModules` vs `modules`
- what `presets` are allowed to bundle
- how `profiles` differ from `presets`
- how explicit composition relates to module-transitive dependencies
- how duplicates should be handled

## Decision

- Label the next API-contract refinement as `v0.2.0`.
- Keep `inputModules` as the raw upstream layer using the existing dotted ref
  format.
- Keep `modules` as the local Semble abstraction layer for cases that need a
  local capability name, local behavior, or `sb.*` options.
- Allow `presets` to bundle:
  - `modules`
  - `inputModules`
  - optional `values`
- Keep `profiles` as broad baselines that compose only `presets`.
- Treat preset-level `inputModules` as valid reusable composition, without
  forcing an immediate Semble module abstraction.
- Resolve explicit composition before transitive module dependencies:
  1. `profiles` expand to `presets`
  2. `presets` contribute explicit `modules`, `inputModules`, and `values`
  3. hosts contribute explicit `modules` and `inputModules`
  4. included Semble modules then contribute transitive `inputs`
- Keep value precedence as:
  1. module defaults and module behavior
  2. preset `values`
  3. host-derived defaults such as `networking.hostName`
  4. host `configFile`
- Keep duplicate inclusion as a hard error across:
  - `modules`
  - `inputModules`
  - `presets`
  - `profiles`
  - overlap between explicit `inputModules` and module-transitive `inputs`
- Require duplicate errors to include provenance, so users can see where each
  duplicate came from and which ownership path to remove.

## Consequences

- `inputModules` and `modules` remain distinct entry layers with different
  responsibilities.
- `presets` become the true reusable composition layer rather than a
  modules-only wrapper layer.
- Consumers can reuse repeated upstream bundles without being forced into fake
  Semble module abstractions too early.
- `profiles` stay focused on broad machine baselines instead of becoming a
  second generic bundle layer.
- Duplicate handling stays strict, which keeps ownership paths clear but makes
  provenance-tracking part of the implementation contract.

## Alternatives Considered

- Require presets to bundle only local Semble modules.
- Remove presets and force repeated bundles into modules or profiles.
- Replace dotted `inputModules` refs with structured objects.
- Allow duplicate imports implicitly as long as the resolved module is the same.

## Notes

This ADR is the current API-contract basis for the next Semble implementation
pass and should be reflected in `docs/api.md`.
